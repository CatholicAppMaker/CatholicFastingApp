# Catholic Fasting App 4.1 Planning Reset

Last reviewed: April 2, 2026
Current live baseline: `4.0 (20)`

This document resets product planning after the successful `4.0` launch. It is intentionally broader than a patch checklist and intentionally less specific than a build-ready feature plan, because the next release theme depends on a small set of anchor features that still need to be chosen.

## Goal

Plan `4.1` as the next meaningful product step after `4.0`, not as a cleanup patch.

`4.1` should combine:

- deeper formation/product capability
- another quality and refinement sweep
- trust and growth improvements around the now-live app
- 2 to 3 explicit anchor features that make the release story clear

## Why 4.1 is a separate planning phase

`4.0` already delivered the major-release bar it was meant to hit:

- materially deeper Canada-aware support
- stronger premium formation workflows
- Guided Seasonal Journey as a clear premium anchor
- Spanish quality cleanup and French Canadian support
- a broad UI, localization, and test hardening pass

That means the next release should not be defined by “more of the same” or by a grab bag of small improvements. `4.1` needs a new release story built around a few chosen product anchors.

## 4.1 Workstreams

### 1. Formation Depth

Build beyond the `4.0` premium/devotional baseline.

Likely themes:

- better daily formation rhythm
- richer reflection and review
- stronger long-term continuity across seasons
- clearer ongoing accountability and progress memory

### 2. Refinement and Stability

Continue tightening the live product where it most matters.

Carry forward:

- localization follow-through
- iPad workflow refinement
- accessibility cleanup
- performance and interaction polish
- any post-launch regressions or user-friction fixes

### 3. Growth and Trust

Finish the public-facing trust layer and improve how the live app explains itself.

Likely themes:

- first-party hosted support, privacy, and terms pages
- stronger onboarding conversion
- clearer premium value communication
- better public-facing support and launch credibility

### 4. New 4.1 Anchor Features

Choose 2 to 3 explicit features and make them the center of the `4.1` story.

This release should not be framed as:

- only cleanup
- only monetization tuning
- only visual polish
- only trust-site work

## What still needs to be decided

The next planning session should lock these choices explicitly:

1. The 2 to 3 anchor features for `4.1`
2. Which one is the headline feature, if any
3. Whether each anchor is free-tier or Premium-only
4. Whether subscription architecture stays unchanged
5. Whether `4.1` is primarily a formation release, a refinement release, or a mixed release with one dominant narrative

Until those are chosen, do not start large implementation work under the label `4.1`.

## Build-Ready Planning Targets

Once the anchors are chosen, the full `4.1` plan should lock:

- release theme and headline value proposition
- affected surfaces on iPhone and iPad
- any new reminder, guidance, tracking, or premium model changes
- localization expectations for English, Spanish, and French Canadian
- trust/growth items that ship with the release
- test and rollout requirements at the same quality bar as `4.0`

## Working Defaults

Unless a chosen anchor feature requires otherwise, assume:

- pricing stays aligned with the live `4.0` configuration
- subscription IDs and group structure stay stable
- the four-tab shell remains intact
- food guidance stays a first-class use case but not a new top-level tab
- `4.1` should be broad and intentional, but still organized around a small number of anchor features

## Next Planning Session Inputs

Come to the next `4.1` planning pass with:

- the candidate feature list
- the preferred headline feature
- any free vs Premium boundaries you already want
- any trust/growth work you consider non-negotiable

That is the minimum input needed to turn this planning reset into a decision-complete implementation plan.
